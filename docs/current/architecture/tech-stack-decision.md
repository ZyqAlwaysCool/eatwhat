# 当前技术栈决策

## 当前确认的技术栈

前端：

- Taro
- React
- TypeScript
- Tailwind CSS

后端：

- Python
- FastAPI

数据层：

- V0 可先使用 SQLite
- 后续稳定迭代建议迁移到 PostgreSQL

部署原则：

- 当前阶段先遵照微信小程序开发要求完成前后端服务开发
- 后端服务默认按可容器化、可迁移的方式实现
- 具体部署承载方式暂不锁定，可在后续于 `阿里云 ECS + AnyService` 与 `微信云托管` 之间再做选择

## 关键决策

### 为什么不是 Next.js 直接做小程序

Next.js 的定位是 Web 全栈框架，不是微信小程序运行载体。

如果目标是微信小程序，React 语法可以保留，但应通过适合小程序的平台层来实现。

### 为什么不是直接使用 shadcn/ui

`shadcn/ui` 更适合普通 Web 运行环境。

当前项目应借鉴其设计思路，而不是直接依赖其组件实现。

### 为什么后端不优先选 Go

当前项目第一阶段的后端重点是：

- 登录
- 数据读写
- 规则承载
- AI 代理

这些任务用 FastAPI 更适合快速验证和持续迭代。

## 当前结论

这套技术栈的目标不是“最强性能”，而是“最快形成可维护、可验证、可迭代的产品骨架”。

当前阶段不把部署平台作为一阶决策，而是优先保证：

- 小程序端能力边界清晰
- 后端接口和用户体系清晰
- 服务具备容器化与迁移能力
