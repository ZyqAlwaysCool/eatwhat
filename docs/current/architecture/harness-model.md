# Harness 控制模型

本文档定义 Agent Harness 在当前仓库中的控制模型。

## 目的

Agent Harness 本身不是业务项目，而是一个受控工程环境，用来帮助 agent 在稳定闭环中工作：

- 读取当前真相
- 在正确边界内实现
- 执行验证
- 检查失败原因
- 仅在结果可检查时交付

## Base

`Base` 是仓库级控制层。

它负责定义：

- 启动顺序
- 仓库地图
- 文档分层
- 稳定脚本入口
- 验证预期
- 不应轻易漂移的硬边界

`Base` 应保持轻量，且尽量与具体技术栈解耦。

## Pack

`Pack` 是技术栈实现层。

它负责定义：

- `app/` 内部推荐结构
- 技术栈相关开发约定
- 技术栈相关验证方式
- 框架级代码组织建议

项目可以在不改变 Base 的前提下替换或扩展默认 Pack。

## Policy

`Policy` 是项目治理层。

它负责定义项目级开关，例如：

- 是否允许自动提交
- 新模块是否必须同步更新文档
- 公共 API 是否必须有测试
- agent 是否可以直接修改脚本

当前模板中的 Policy 主要定义在 `configs/agent-harness.yaml`。

## 当前默认配置

当前默认配置为：

- Base：本模板定义的仓库控制规则
- Pack：`taro-react-fastapi`
- Policy：`configs/agent-harness.yaml` 中的值

## 设计意图

这个控制模型遵循 harness engineering 原则：

- 仓库应当能被 agent 读成一个可操作的控制面
- 执行入口应保持稳定
- 验证应默认进入工作闭环
- 临时记录不应与当前真相混在一起
- 人类只应在高影响决策点被拉入
