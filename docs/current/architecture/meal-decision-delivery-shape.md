# 今天吃什么落地形态说明

## 目标

定义当前阶段最可落地的交付形态，避免产品讨论脱离实现现实。

## 当前结论

如果以“先做出可验证产品”为目标，第一阶段更适合做轻量移动端产品，而不是重投入的复杂应用。

当前更推荐的路径是：

- 前端形态确定为微信小程序优先
- 第一版以轻交互、轻视觉资产为原则
- 后端不是第一天就必须存在，但应为后续扩展预留空间

## 前端技术方向建议

如果确定交付形态是微信小程序，前端更适合采用“React 语法的小程序框架”，而不是直接采用 Next.js 作为小程序运行载体。

当前建议方向：

- 小程序端：React 风格开发
- 跨端或编译层：优先考虑 Taro 一类支持 React 语法并输出微信小程序的平台
- Next.js：如有需要，可用于未来官网、活动页、管理后台或 Web 端，不建议作为小程序客户端主框架

这样做的原因是：

- 小程序本身不是普通 Web 运行环境
- Next.js 的核心定位是 Web 全栈应用框架，不是微信小程序编译目标
- 如果希望保留 React 开发体验，同时落地小程序能力，跨端小程序框架更合适

## 为什么优先轻量形态

- 使用频次高，但单次使用时长短
- 用户目标非常明确，不需要复杂浏览路径
- 你当前不希望依赖复杂美术资产，这与轻量形态匹配
- 第一阶段更重要的是验证决策链路，而不是做完整平台

## 第一版最小实现建议

推荐的第一版实现方式：

- 单人场景优先
- 用户手动维护品类与店铺候选池
- 本地保存或轻后端保存基础数据
- 使用结构化标签、规则过滤和加权抽选完成决策
- 记录接受、跳过、近期吃过等反馈

## 第一版架构约束

为了支持从 V0 平滑升级到 V1，当前建议采用以下架构约束：

- 页面层、状态层、决策规则层分离
- 标签定义与筛选规则分离
- 候选池数据模型与抽选算法分离
- 单人决策流程与多人协同流程分离
- AI 辅助能力与基础规则链路分离

这样做的目的，是避免第一版写成难以升级的耦合实现。

## 标签与规则的实现原则

第一版建议优先使用结构化标签，而不是依赖自然语言理解。

建议分层：

- 内置标签：参与稳定筛选和排序
- 用户自定义标签：第一版先用于补充描述，不强依赖核心算法
- 场景标签：例如家附近、公司附近、周末可去、外卖常点

建议的决策链路：

1. 根据硬条件过滤候选项
2. 根据软偏好和历史反馈进行轻量打分
3. 在高分候选中执行加权抽选
4. 生成结果并记录本轮反馈

## 录入交互原则

第一版的录入体验必须比管理待办事项更轻。

建议原则：

- 新建店铺时只要求最少字段
- 允许“先抽品类，后补店铺”
- 支持在结果页一键收藏为常用候选
- 优先通过使用沉淀数据，而不是通过初始化表单索取数据

## 视觉与交互原则

小程序设计语言应保持简洁、干净、聚焦。

建议原则：

- 减少装饰性元素
- 优先使用清晰卡片、标签、分组列表
- 控制动画数量，只在出结果等关键节点使用
- 避免转盘类强游戏化视觉成为主界面中心
- 保持核心按钮层级明确，例如“开始”“换一个”“加入候选池”

当前判断是：

- 第一版需要一定的动效和反馈感
- 这些动效完全可以通过常规前端动画能力完成
- 不需要为了第一版额外引入 AI 生图工作流

更适合的做法是：

- 使用基础过渡动画
- 使用卡片切换、数字滚动、轻量揭晓效果
- 保持视觉系统克制，避免对复杂素材的依赖

## 后端什么时候变成必需

在以下需求出现后，后端会从“可选”变成“必要”：

- 需要跨设备同步
- 需要双人或多人共享候选池
- 需要账号体系
- 需要位置与附近店铺能力
- 需要接入外部 POI 或地图数据
- 需要更复杂的推荐策略

## 对模板仓库的含义

当前仓库的 `python-fastapi` Pack 仍然可以保留，因为它适合作为后端能力和控制面的基础。

但如果项目继续推进，模板最终不应只描述 FastAPI 单体应用，而应逐步演进为：

- 轻前端交付形态
- 可选 Python API 能力
- 分阶段增强的脚本与验证入口

## 当前建议

既然第一版载体已确定为微信小程序，下一步应围绕以下事项继续收紧：

1. 页面流与低保真原型
2. 标签体系与规则算法
3. 候选池录入流程
4. 模板中的 Pack、脚本和验证入口调整
5. 小程序技术栈与工程结构选择

## 当前部署原则

当前阶段先不锁定后端部署承载方式。

先确认和实现的内容是：

- 小程序端页面与交互
- 前后端接口契约
- 用户登录与数据模型
- 后端服务的容器化能力

后续再根据实际开发情况和接入约束，在以下方案中做选择：

- `阿里云 ECS + AnyService`
- `微信云托管`

这样可以避免在产品和工程骨架尚未稳定前，把过多精力消耗在部署平台选择上。
