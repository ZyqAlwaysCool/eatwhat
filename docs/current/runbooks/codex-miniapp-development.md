# Codex 小程序开发运行手册

## 目标

说明如何在本地依靠 Codex 完成微信小程序项目开发。

## 推荐工具组合

- VS Code：主要编码环境
- Codex：主要编码 agent
- 终端：执行脚本、构建、测试
- 微信开发者工具：小程序预览、模拟器与真机测试

## 推荐流程

1. 使用 Taro 初始化项目
2. 使用 VS Code 打开仓库
3. 让 Codex 在仓库内完成页面、组件、规则和后端开发
4. 在 `app/api` 中用 `uv` 管理 Python 依赖与虚拟环境
5. 后端按容器化标准实现与验证
6. 在终端运行验证命令
7. 使用微信开发者工具导入 `app/miniapp` 项目根目录
8. 在开发者工具中查看页面效果和授权行为
9. 在需要时做真机测试

## 当前阶段的部署约束

在当前阶段，重点是把前后端服务开发完成，而不是立即锁定部署平台。

因此默认要求是：

- 后端具备 Docker 化能力
- 配置通过环境变量管理
- 服务健康检查和启动命令明确
- 部署承载方式后置决策

## 与 iOS 工作流的对应关系

可以把它理解为：

- Taro 脚手架 ≈ Xcode 创建项目
- VS Code + Codex ≈ 主要编码与编译修复环境
- 微信开发者工具 ≈ 小程序平台侧模拟器和调试工具

## Codex 在这个流程中的职责

Codex 负责：

- 编写代码
- 调整结构
- 运行 lint、测试、类型检查
- 修复构建问题
- 补充文档与注释

开发者负责：

- 决定产品边界
- 确认平台授权策略
- 在微信开发者工具中确认最终效果
- 进行真机体验判断

## 关键提醒

- 小程序平台行为不能只靠静态阅读判断
- 最终效果仍应在微信开发者工具中确认
- 平台限制相关问题需要优先做小范围验证
- 当前仓库前端建议使用 `Node.js 20 LTS`，后端依赖管理使用 `uv`
- 当前仓库默认运行入口为根脚本：`./scripts/setup`、`./scripts/dev`、`./scripts/migrate`、`./scripts/check`
- 如果要单独进入后端目录执行命令，请以 `app/api` 作为 `uv` 项目根
