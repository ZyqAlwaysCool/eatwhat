# 今天吃什么产品探索记录

日期：2026-03-29

## 用户问题摘要

当前提出的核心痛点是：

- “今天吃什么”会反复出现，不是低频问题
- 用户通常不是完全没有想法，而是有多个候选项却难以收敛
- 决策链路往往被拆散在多个地方，例如脑内记忆、美团、大众点评、高德等
- 真正消耗时间的是反复比较与反复否决，而不是发现选项本身

## 初步判断

这个问题真实存在，也足够常见，具备做成产品探索的价值。

但这个方向不能简单理解为“做一个随机转盘”。如果产品只提供随机能力，差异性会很弱，也很容易被替代。

## 竞品与替代品快照

### 1. 通用随机决策产品已经存在

可见样本：

- `随机轮盘-选择困难症做决定`
- `小决定 - 选择困难症救星`

判断：

- 随机转盘机制本身不新
- 纯随机工具开发门槛低，但护城河也低

### 2. 吃饭垂类随机工具也已经存在

可见样本：

- `吃什么 - 选择困难症克星`

判断：

- “帮用户决定吃什么”本身已经有人在做
- 只做“附近随机选一家”未必足够强

### 3. 双人或情侣决策方向也已经有人尝试

可见样本：

- `What to Eat? For Couples`

判断：

- 多人协同吃饭决策是可被单独包装的需求
- 但并不意味着项目必须从双人切入

### 4. 强替代品来自已有大平台

可见样本：

- `大众点评`
- `高德地图`
- `美团外卖`

判断：

- 用户并不缺店铺信息
- 用户缺的是“从大量选项中快速收敛出一个可执行结果”的过程

## 可行性分析

### 问题可行性

问题本身成立，且高频重复出现，适合做轻量工具型产品。

### 产品可行性

要想落地，第一版必须缩短链路，不能上来就做重平台。

更现实的最小可行产品路径是：

1. 用户维护有限候选池
2. 系统根据简单条件过滤
3. 系统给出一个明确结果
4. 用户通过接受或跳过反馈系统

### 差异化可行性

更可能成立的差异化点：

- 用户私有候选池
- 更快的决策收敛流程
- 品类和店铺的连续决策
- 后续可扩展到双人或多人协同

不太成立的方向：

- 只有一个随机轮盘

## 当前建议的产品切入

建议先从单人场景切入，而不是先做双人。

原因：

- 实现复杂度更低
- 更容易验证真实使用频率
- 不需要一开始就解决协同、公平性、共享数据等问题
- 如果单人版本都无法形成价值，多人版本通常也不会成立

单人版本跑通之后，再扩展到双人或多人，整体风险更低。

## 当前建议的交付形态

从落地效率看，首发形态更适合是轻量移动端产品，例如微信小程序，而不是重投入的原生 App。

原因：

- 使用场景高频但停留时长短
- 更适合快速验证而不是重资产打磨
- 更符合“弱视觉依赖”的现实约束

这仍然是当前建议，不是最终锁定结论。

## 当前建议的技术落地路径

如果以可行性为第一原则，第一版不应依赖复杂外部能力。

建议优先顺序：

1. 手动录入候选品类
2. 手动录入候选店铺
3. 本地或轻后端保存数据
4. 简单规则抽选
5. 结果反馈回收

不建议第一版就做：

- 第三方平台深度抓取
- 重推荐算法
- 复杂地图能力
- 大规模账号体系

## 对当前模板仓库的启发

当前仓库默认是 `python-fastapi` 模板，这对后端和控制面仍然有价值，但它不一定等于最终交付形态。

如果项目继续推进，大概率需要从“纯 FastAPI 示例模板”演进成“轻前端 + 可选 Python API”的项目模板。

这意味着后续很可能需要：

- 新的 Pack 说明
- 更细的脚本约束
- 前后端分离的验证入口

## 待确认问题

- 第一版以到店吃饭为主，还是同时覆盖外卖
- 第一版是否只依赖用户手动录入候选池
- 第一版推荐结果是否必须包含具体店铺
- 后续扩展时，多人上限应该是 2 人、4 人，还是不预设
- 核心成功标准究竟是“更快决定”，还是“决定质量更高”
