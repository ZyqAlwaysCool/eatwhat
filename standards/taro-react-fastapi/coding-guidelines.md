# 编码规范

## 总体原则

- 代码以可维护性为第一原则
- 默认选择主流、稳定、可理解的写法
- 不为了“高级感”引入难维护的技巧
- 一个模块只承担清晰责任

## 可读性要求

- 命名要直接表达意图
- 函数长度保持可控
- 避免超长文件和超深嵌套
- 相邻代码应体现单一职责

## 注释要求

- 关键代码片段需要增加简要中文注释
- 注释应解释“为什么这样做”或“这里的业务边界是什么”
- 不写机械重复代码含义的废话注释
- 简单赋值、简单条件判断通常不需要注释

推荐场景：

- 抽选算法中的权重规则
- 标签筛选与历史反馈的耦合逻辑
- 微信小程序平台限制相关处理
- 登录态与用户绑定流程

## React 代码要求

- 使用函数组件
- 使用 TypeScript 明确 props 和关键数据结构
- 页面层避免堆积业务判断
- 可复用逻辑优先提取成 hooks 或业务模块
- 避免不必要的过度抽象

## FastAPI 代码要求

- 路由、schema、service 分层明确
- 不在路由里写复杂业务逻辑
- 不把数据库细节泄露到上层服务
- 关键业务规则应可单测

## 样式与组件要求

- 优先复用组件，不要重复堆砌相似页面结构
- 样式保持统一节奏和视觉秩序
- 避免随意使用硬编码颜色和间距

## 测试要求

- 新增公共行为时需要对应测试
- 复杂规则模块优先补单元测试
- 页面关键交互至少应具备可验证路径

## Codex 编码行为要求

- 改动前先理解现有边界
- 优先在既有结构内扩展，而不是随意重组
- 新增模块时同步补必要文档
- 提交前说明运行过哪些验证
